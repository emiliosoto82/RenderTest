# Chaos API – Entrenamiento de mantenimiento real

Este repositorio NO es un ejercicio de programación clásico.
NO es un tutorial.
NO es un proyecto “bonito”.

Es un **entrenamiento de mantenimiento de software en condiciones reales**.

Aquí vas a trabajar con:
- código escrito por otras personas
- decisiones poco claras
- errores razonables
- cambios que parecen correctos pero chocan entre sí
- situaciones donde no tienes toda la información

Exactamente como ocurre en un entorno profesional.

---

## Qué es este proyecto

Chaos API es una API sencilla en .NET que funciona como soporte para un conjunto
de **misiones de trabajo real**.

El objetivo principal NO es:
- aprender .NET
- aprender Git como lista de comandos
- escribir código perfecto

El objetivo es aprender a:
- **entender sistemas que evolucionan mal**
- **integrar trabajo ajeno**
- **resolver problemas sin romper lo existente**
- **asumir decisiones técnicas sin certezas**
- **no entrar en pánico cuando algo no encaja**

---

## Qué vas a vivir aquí

A lo largo de las misiones:

- El sistema empezará funcionando bien
- Varias personas lo modificarán sin coordinarse
- Aparecerán incoherencias
- El código seguirá compilando… pero no siempre tendrá sentido
- Habrá presión, urgencias y decisiones heredadas
- Te tocará arreglar cosas que tú no rompiste

Todo esto es **intencionado**.

Si algo parece caótico, **es parte del entrenamiento**.

---

## Qué NO es este repositorio

Este repositorio NO es:

- ❌ un examen
- ❌ una kata
- ❌ un proyecto para lucirse
- ❌ un sitio donde borrar y empezar de cero

Aquí **no vale**:
- borrar funcionalidad “para que funcione”
- volver al estado inicial
- tomar atajos por reflejo
- preguntar qué quiso hacer otra persona

---

## Reglas generales (muy importantes)

✅ Puedes:
- usar Visual Studio o VS Code
- usar terminal
- buscar documentación o información externa
- equivocarte y corregirte

❌ No puedes:
- reiniciar el proyecto
- crear un repo nuevo
- borrar funcionalidades existentes
- preguntar a otras personas qué pretendían
- asumir que algo es “basura” sin entenderlo

Asume siempre esto:
> **Todo lo que existe en el sistema está ahí por una razón.  
> Puede no ser buena, pero existe.**

---

## Cómo se trabaja

- Cada misión está descrita en la carpeta `/missions`
- Las misiones se hacen **en orden**
- Normalmente trabajarás en ramas propias
- Integrarás trabajo de otras personas
- No siempre arreglarás lo que tú hiciste

Después de cada misión:
- deberás completar un **POST‑MORTEM**
- no para justificarte
- sino para **entender qué pasó y qué aprendiste**

---

## Sobre el POST‑MORTEM

El post‑mortem es parte fundamental del entrenamiento.

No es un trámite.
No es teoría.
No es “poner lo que quedó bien”.

Es el momento donde conviertes experiencia en criterio.

Si te lo tomas en serio, este repositorio te hará mejorar más que muchos cursos.

---

## Qué se espera de ti

No se espera perfección.
No se espera saberlo todo.
No se espera rapidez.

Se espera que:
- pienses antes de tocar
- leas código ajeno con respeto
- asumas decisiones
- no rompas por reflejo
- dejes el sistema mejor que como lo encontraste

---

## Aviso final

Si en algún momento piensas:

Entonces has aprendido lo importante.
